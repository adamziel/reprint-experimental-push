## Findings by lane

### 1. release-gates

**What held:** The checked release command fails closed with no live source,
prints the GATE-3 topology evidence, rejects packaged fallback release movement,
and keeps `gates: "0/4"`.

**False or unsafe claim to prevent:** Local verifier output and progress pages
still contain `releaseMovement.allowed: true` / `gates: candidate-for-review`
for local Playground topologies. That label is acceptable only if every consumer
keeps it distinct from final release movement. It is unsafe if a status page,
release gate evaluator, or operator runbook treats `candidate-for-review` as a
partially open release gate.

**Updated evidence:** After rebasing, `test/release-gates.test.js` passes 8/8.
The new evaluator does distinguish local candidate evidence from final release
readiness.

**Remaining missing test:** The evaluator still needs to be fed by the actual
required-check results and progress artifacts, not only synthetic summaries. It
must fail the final verdict when `npm test` or another required suite is red.

**Exact next correction:** Wire the release-gate evaluator into the public
progress/reporting path and the required-check list. Assert that all local
`candidate-for-review` proofs are downgraded to `support_only`, and that red
required checks force the overall release verdict to `blocked` or `0/4`
regardless of local receipts.

### 2. executor-auth-leases

**What held:** Many low-level auth/session and signed-request tests pass, and
the live release verifier preserves the explicit-source fail-closed behavior.

**Blocking failures observed:**

- `test/production-shaped-proof.test.js` lifecycle summary helper tests fail in
  isolation. The evaluator reports `stale-issued-summary` before it reaches
  expected preserved-read or authenticated-identity failures. The mismatch comes
  from comparing a compact `summary.issued` object against normalized trace
  observations that include fields such as `step` and `preserved`.
- `test/authenticated-http-push-client.test.js` has 10 failures in preserved
  auth/session lifecycle and lease/retry boundary behavior. Examples include
  revoked/cleaned-up lifecycle observation recording stopping after dry-run,
  preserved-remote retry precedence swapping between
  `PRESERVED_REMOTE_RETRY_REQUIRED`, `DURABLE_JOURNAL_NOT_PROVEN`, and
  `PRODUCTION_AUTH_SESSION_LIFECYCLE_REQUIRED`, and missing consumed-claim
  identity preservation after a validated production recovery journal surface.

**False or unsafe claim to prevent:** Do not claim production auth/session or
lease fencing is release-ready while the auth client file and lifecycle summary
proof file are red. The current evidence is support-level, not a GATE-1 proof.

**Exact next correction:** Normalize `issued` and `read` summaries before
comparing them with trace observations, or require the summary schema to carry
the same normalized fields. Then decide and codify the precedence order among
preserved-remote retry, durable-journal proof, and production auth/session
lifecycle failures. Re-run `node --test test/authenticated-http-push-client.test.js`
and the production-shaped lifecycle summary tests before claiming any gate
movement.

### 3. journal-recovery

**What held:** Focused model recovery and journal tests passed: stale recovery
claims fence old workers, partial mutation classifies as blocked recovery,
completed replays do not duplicate inserts, and the checked durable journal
boundary stays closed until stale-claim rejection is proven.

**Missing production proof:** These passing tests do not erase the failing auth
client boundary tests, because the release path must consume journal, lease,
auth/session, and preserved-remote retry evidence together. The full suite red
status means GATE-2 remains `support_only` at best.

**Exact next correction:** Keep the journal-recovery lane scoped to a single
release-path boundary object that includes paged readback, consumed claim id,
stale-owner rejection, preserved-remote retry evidence, and auth/session
readback. Add a regression that fails if the recovery-journal fallback hides a
missing stale-claim retry proof.

### 4. graph-identity

**What held:** Focused planner graph tests passed for same-plan closures and
fail-closed unsupported surfaces. The graph inventory command emits deterministic
machine-readable evidence.

**Unsafe broadening to prevent:** `blockedFamilies: []` in the benchmark
inventory only means zero blocked families among the 7 benchmark families. It
does not mean the WordPress graph identity problem is solved. The same inventory
reports only one guarded family (`unsupportedPluginOwnedSurfaces`), while the
planner still intentionally blocks menu item, `nav_menu`, and custom taxonomy
surfaces. The mapping is still same-plan/stable-ID closure evidence; it is not a
general ID allocation or reference-rewrite engine.

**Exact next correction:** Make graph inventory counters enumerate every
supported relationship type and every intentionally blocked surface separately.
Rename or annotate `mappedFamilies` so it cannot be read as universal identity
mapping. Add inventory assertions for `nav_menu`, custom taxonomy, and any
serialized block/menu references that remain unsupported.

### 5. plugin-driver

**What held:** The focused plugin-driver boundary tests passed, including
unknown plugin-owned resource blocking, exact owner/driver allowlist matching,
direct `active_plugins` rejection, and unowned serialized option rejection.

**Blocking failures and scope limits:**

- Full `npm test` shows `production plugin package smoke supports
  driver-guard-only mode` failing because the test asserts an exact source-code
  shape (`if (!runDriverGuardOnly)`) that no longer exists.
- `test/playground-snapshot-lib.test.js` fails the forms-lab custom-table guard
  with `Call to undefined function apply_filters()`, so the standalone guard
  path is not loading or shimming the WordPress filter API it expects.
- The release-state plugin-driver proof covers one custom table row:
  `row:["wp_reprint_push_release_state","state_id:1"]`. It is good support
  evidence, but it does not prove arbitrary plugin semantics, activation/update
  rollback, generated files, cron/cache side effects, WooCommerce HPOS, or
  broad custom-table ownership.

**Exact next correction:** Replace source-shape regex assertions with behavior
checks that execute driver-guard-only mode and parse its summary. Fix the
snapshot-lib standalone guard by loading a minimal WordPress filter shim or by
making the test enter through a real WP-loaded path. Keep the release-state row
labeled as one driver proof until at least one real plugin-owned table/option
contract proves export, validate, apply, rollback/block, and redacted audit
behavior.

### 6. chunking-benchmark

**What held:** Benchmark and performance model tests passed. The benchmark
properly refuses production throughput claims and keeps chunks staged until a
guarded publish step.

**Missing production proof:** This is model/local benchmark evidence. It does
not measure production storage receipts, real HTTP chunk cursors, restartable
transfers, or DB batch execution on a non-lab source.

**Exact next correction:** Keep `productionThroughput: not-claimed` until the
release verifier can attach real chunk receipts, storage receipts, recovery
inspection, and memory/backpressure evidence from a production-owned source
run. Add a CI check that fails if benchmark output is used to set a production
throughput claim without those receipts.
